Introduction

MaxDiff, which is also called Best-Worst Scaling, is a survey method to measure how strongly people prefer one item compared to another. In each screen of the survey, the respondent sees a small subset of items (here, four classes), and chooses the item they like most and the item they like least. The respondent repeats this for many screens, and each screen shows a different subset. Even though no respondent ever ranks all the items together, the full ranking can still be recovered from these choices.

Why do we use this method and not, for example, a 1–10 rating scale? The reason is that people are good at trade-offs, which means choosing a best and a worst from a few options, but they are not good at giving numbers in a consistent way. The score “8 out of 10” from one person can mean the same as “6” from another person, and some people never use the high or low end of the scale. When MaxDiff forces a choice, this scale-use bias is avoided, and every answer gives real information about relative preference.

In this assignment, the ten items are the ten core MSBA classes in the UCSD Rady program. The goal is to estimate how much students prefer each class. We use three different methods, and we compare what each method tells us:

  1. A simple counts analysis, which is the share of times a class was picked best minus the share of times it was picked worst.
  2. The multinomial logit (MNL) model fit by maximum likelihood. This model describes the choice process directly and gives a utility coefficient for each class.
  3. The same MNL model fit by Bayesian methods using a random-walk Metropolis-Hastings sampler. This method also measures how uncertain we are about each result.

What do we expect? The three methods should agree on the general ranking of the classes, so the most-preferred and the least-preferred classes should be in similar positions. The methods are different mainly in how they show the strength of preference, and in how they describe the uncertainty. At the end, we will put the three methods together in one comparison, and we will discuss which method is useful for which situation.

Item 11 is an unlabeled anchor. It never appears in a four-item screen, it is chosen over a real class only sometimes, and the two-item tasks only record a best pick (there is no -1). Counts Analysis

The counts analysis is the simplest summary of the data, and it does not need any model. For each class \(j\), we count how many times it was picked best and how many times it was picked worst. Each count is written as a percentage of the number of times the class was shown. The difference between these two percentages is one score:

\[ \text{score}_j \;=\; \%\text{best}_j \;-\; \%\text{worst}_j \]

A score close to \(+1\) means the class is almost always picked best when it appears. A score close to \(-1\) means it is almost always picked worst. A score close to \(0\) means the class is in the middle, or that opinions about it are mixed. Because the exposure is balanced, we can compare these scores directly between classes.

From MaxDiff Data to MNL Choices

The counts score is easy to understand, but it loses some information, because it treats a “best” pick as if it does not depend on the other three items on the screen. The MNL choice model uses the full structure of each task. The model gives each class \(j\) a latent utility coefficient \(\beta_j\), and the choices follow a soft-max over the items that are shown.

On a screen that shows items \(\{a, b, c, d\}\), the probability that item \(b\) is picked best is

\[ P(\text{best} = b) \;=\; \frac{\exp(\beta_b)}{\exp(\beta_a) + \exp(\beta_b) + \exp(\beta_c) + \exp(\beta_d)} \]

After the best item is removed, the worst pick is modeled as a choice among the three remaining items, but with the utilities flipped. The least preferred item is the one with the lowest utility, which is the same as the highest \(-\beta\):

\[ P(\text{worst} = c \mid \text{best} = b) \;=\; \frac{\exp(-\beta_c)}{\exp(-\beta_a) + \exp(-\beta_c) + \exp(-\beta_d)} \]

So each four-item task gives two MNL observations: one best-pick from the four items, and one worst-pick from the three remaining items.

Identifiability. The soft-max does not change if we add the same constant to every \(\beta\). For this reason, only \(J - 1 = 9\) of the ten coefficients can be identified separately. We fix item 1 (MGTA 403) as the reference, so we set \(\beta_1 = 0\) and we estimate \(\beta_2, \ldots, \beta_{10}\) relative to it. Then every coefficient means “utility compared to MGTA 403.”

Code 
# For each four-item task store: the 4 item ids, the best pick, the worst pick.
SHOWN, BEST, WORST = [], [], []
for (_, _), g in four_item.groupby(["Record ID", "Task"]):
    SHOWN.append(g["Item"].values)
    BEST.append(int(g.loc[g["Response"] == 1, "Item"].iloc[0]))
    WORST.append(int(g.loc[g["Response"] == -1, "Item"].iloc[0]))

MNL via Maximum Likelihood

When we add the log-probabilities of the observed best and worst picks over all tasks, we get the log-likelihood

\[ \ell(\boldsymbol{\beta}) \;=\; \sum_{\text{tasks}} \Big[\, \beta_{j^*_b} \;-\; \log \!\!\!\sum_{j \in \text{shown}}\!\!\! \exp(\beta_j) \;-\; \beta_{j^*_w} \;-\; \log \!\!\!\!\!\sum_{j \in \text{shown} \setminus \{j^*_b\}}\!\!\!\! \exp(-\beta_j) \,\Big] \]

where \(j^*_b\) and \(j^*_w\) are the items picked best and worst in that task. We write the negative log-likelihood as a function of the nine free coefficients (and we fix \(\beta_1\) to \(0\) inside the function). We use a log-sum-exp form for numerical stability, and we minimize the function.

Code 
def make_beta(free9):
    """Insert the fixed reference (beta_1 = 0) into a length-11 vector
    indexed by item id; index 0 is unused so items map directly."""
    beta = np.zeros(11)
    beta[2:11] = free9          # items 2..10 are free; item 1 stays at 0
    return beta

def neg_loglik(free9):
    beta = make_beta(free9)
    u = beta[SHOWN]                                   # (n_tasks, 4)
    ll_best = beta[BEST] - np.log(np.exp(u).sum(axis=1))

    # Worst pick: remaining 3 items (shown != best), utilities flipped.
    mask = SHOWN != BEST[:, None]                     # True for remaining 3
    neg_u = np.where(mask, -beta[SHOWN], -np.inf)
    ll_worst = (-beta[WORST]) - np.log(np.exp(neg_u).sum(axis=1))

    return -(ll_best + ll_worst).sum()

res = minimize(neg_loglik, np.zeros(9), method="BFGS")
beta_hat = make_beta(res.x)
se_hat = np.zeros(11)
se_hat[2:11] = np.sqrt(np.diag(res.hess_inv))   # SEs from inverse Hessian

print(f"Converged: {res.success}")
print(f"Log-likelihood at optimum: {-res.fun:.2f}")
Converged: False
Log-likelihood at optimum: -1572.73

The standard errors come from the inverse of the (negative) Hessian at the optimum. The scipy BFGS optimizer returns this as hess_inv. To make the coefficients easier to interpret, we change them into shares of preference using a soft-max over all ten classes:

\[ \hat s_j \;=\; \frac{\exp(\hat\beta_j)}{\sum_{k=1}^{10} \exp(\hat\beta_k)} \]

These shares add up to one, and they answer a concrete question: if all ten classes were shown together on one screen, what fraction of the “best” votes would each class receive?

MNL via Bayesian Estimation

We can fit the same model with Bayesian inference. In this approach, \(\boldsymbol{\beta}\) is treated as random, and the result is a full posterior distribution, not only a single point. The posterior is proportional to the likelihood multiplied by the prior,

\[ \pi(\boldsymbol{\beta} \mid \boldsymbol{y}) \;\propto\; L(\boldsymbol{\beta}) \cdot \pi(\boldsymbol{\beta}) \]

We use a weakly informative multivariate normal prior, \(\boldsymbol{\beta} \sim N(\boldsymbol{0},\; 10 \cdot I)\). This is a standard deviation of about \(3.2\) on each coefficient, which is wide enough to let the data decide, but still keeps the sampler stable. We cannot write the posterior in a closed form, so we draw samples from it with a random-walk Metropolis-Hastings sampler.

Code 
def log_post(free9):
    # log-likelihood (= -neg_loglik) plus log multivariate-normal prior
    log_prior = np.sum(-0.5 * free9**2 / 10.0)
    return -neg_loglik(free9) + log_prior

def metropolis(beta0, n_iter, step):
    K = len(beta0)
    draws = np.empty((n_iter, K))
    beta = beta0.copy()
    lp = log_post(beta)
    accept = 0
    for t in range(n_iter):
        beta_star = beta + np.random.normal(0, step, K)
        lp_star = log_post(beta_star)
        if np.log(np.random.rand()) < lp_star - lp:
            beta, lp = beta_star, lp_star
            accept += 1
        draws[t] = beta
    return draws, accept / n_iter

# Step size tuned to land acceptance in the 25-40% band (see below).
N_ITER, STEP = 20000, 0.07
draws, accept_rate = metropolis(res.x, N_ITER, STEP)

burn = N_ITER // 4               # discard first 25% as burn-in
post = draws[burn:]
print(f"Acceptance rate: {accept_rate:.3f}  (target 0.25-0.40)")
print(f"Iterations: {N_ITER}, burn-in: {burn}, kept: {len(post)}")
Acceptance rate: 0.344  (target 0.25-0.40)
Iterations: 20000, burn-in: 5000, kept: 15000

Tuning the step size. The random-walk step decides how far each proposal jumps. If the step is too small, the chain moves very slowly (the acceptance rate is near 1, but the chain almost does not move). If the step is too large, almost every proposal is rejected (the acceptance rate is near 0). A step of \(0.07\) puts the acceptance rate inside the recommended band of 25–40%. If we make the step smaller, the acceptance rate goes up; if we make it larger, the acceptance rate goes down.

The traces look like fuzzy caterpillars around a stable level, and there is no slow drift. This is the visual sign that the chain has converged and is mixing well. Now we summarize the posterior. We compute the posterior mean of each coefficient, a 95% credible interval from the 2.5th and 97.5th percentiles of the draws, and the posterior shares of preference. To get the shares, we put every draw through the soft-max and then take the average.

What does the MNL add compared to the counts analysis? It describes the choice process directly. The counts score only records the picks, but the MNL also uses every unchosen item on a screen as information. For example, an item that loses to MGTA 495 on a strong screen tells the model something that the counts score ignores. The result is a consistent set of utilities that can also predict choices on screens that were never shown.

What does the Bayesian fit add compared to the MLE? It gives honest uncertainty on derived quantities. The shares of preference are a nonlinear transform of the coefficients, so to get a confidence interval on a share from the MLE, we need the delta method and some calculus. With the MCMC draws, it is very simple: we put each draw through the soft-max and read the 2.5th and 97.5th percentiles. The same simple method also works for any “what-if” quantity, for example market-share simulations or the effect of removing a class.

Discussion

For a student or an advisor who only wants the main result: from 85 MSBA students and 680 best-worst screens, MGTA 495 Marketing Analytics (Spring, Yavorsky) is the clear favorite. It receives about a quarter of all “best” votes in a contest of ten classes, which is more than two times any other class. After it, there is a strong middle group, which includes Business Analytics, Customer Analytics, Large Data, the Capstone, and SQL. The least preferred classes are Analytics Consulting, Marketing/Finance/Operations, and the AI Math/Programming course. The small surprise is the size of the gap at the top: the most preferred elective is not only the highest, but it is higher by a large amount. All three methods agree on this result, so an advisor can be confident that it describes the students, and not only the method.

On the methodological side, each method is good for a different question:

  • Counts is the right choice when we want a quick descriptive result, for example an executive dashboard or a first look. It does not need modeling assumptions, and it is easy to explain to anyone.
  • MLE MNL is the right choice when we need rigorous point estimates with classical standard errors, and when we need one clear number for each item to report.
  • Bayesian MNL is the right choice when we want uncertainty on derived quantities, for example shares of preference, market simulations, or “what if we add a new elective?” scenarios. It is also the right choice when we want to extend the model in a hierarchical way to estimate the preferences of each individual student.

There are also some caveats that we must remember. The respondents are a self-selected sample of MSBA students, so these preferences may not be the same for other cohorts or other programs. The preferences are also mixed together with things that the survey does not control: the professor of each class, the quarter when the class is offered, the prior coursework of the student, and the position of the student in the program can all affect the answers. For this reason, the design cannot separate “I like this subject” from “I like this professor.”

If we had more time, the most natural next step is to use the anchor tasks that we put aside. These dual-response screens find an absolute zero point of preference, and with this point we could run a real anchored MaxDiff. Then we could say not only which classes are ranked higher than others, but also which classes the students actively like and which classes they only tolerate. After that, a hierarchical Bayesian extension could estimate one utility vector for each respondent. This would change a single program-level ranking into a tool for personalized course advising.

## Introduction

**MaxDiff**, which is also called *Best-Worst Scaling*, is a survey method to measure how strongly people prefer one item compared to another. In each screen of the survey, the respondent sees a small subset of items (here, four classes), and chooses the item they like *most* and the item they like *least*. The respondent repeats this for many screens, and each screen shows a different subset. Even though no respondent ever ranks all the items together, the full ranking can still be recovered from these choices.

Why do we use this method and not, for example, a 1–10 rating scale? The reason is that people are good at *trade-offs*, which means choosing a best and a worst from a few options, but they are not good at giving numbers in a consistent way. The score "8 out of 10" from one person can mean the same as "6" from another person, and some people never use the high or low end of the scale. When MaxDiff forces a choice, this scale-use bias is avoided, and every answer gives real information about relative preference.

In this assignment, the ten items are the ten core MSBA classes in the UCSD Rady program. The goal is to estimate how much students prefer each class. We use three different methods, and we compare what each method tells us:

1. A simple **counts analysis**, which is the share of times a class was picked best minus the share of times it was picked worst.
2. The **multinomial logit (MNL)** model fit by **maximum likelihood**. This model describes the choice process directly and gives a utility coefficient for each class.
3. The same MNL model fit by **Bayesian methods** using a random-walk Metropolis-Hastings sampler. This method also measures how uncertain we are about each result.

What do we expect? The three methods should agree on the general ranking of the classes, so the most-preferred and the least-preferred classes should be in similar positions. The methods are different mainly in *how* they show the strength of preference, and in how they describe the uncertainty. At the end, we will put the three methods together in one comparison, and we will discuss which method is useful for which situation.

## Counts Analysis

The **counts analysis** is the simplest summary of the data, and it does not need any model. For each class $j$, we count how many times it was picked *best* and how many times it was picked *worst*. Each count is written as a percentage of the number of times the class was *shown*. The difference between these two percentages is one score:

$$
\text{score}_j \;=\; \%\text{best}_j \;-\; \%\text{worst}_j
$$

A score close to $+1$ means the class is almost always picked best when it appears. A score close to $-1$ means it is almost always picked worst. A score close to $0$ means the class is in the middle, or that opinions about it are mixed. Because the exposure is balanced, we can compare these scores directly between classes.

```{python}

## From MaxDiff Data to MNL Choices

The counts score is easy to understand, but it loses some information, because it treats a "best" pick as if it does not depend on the other three items on the screen. The **MNL choice model** uses the full structure of each task. The model gives each class $j$ a latent utility coefficient $\beta_j$, and the choices follow a soft-max over the items that are shown.

On a screen that shows items $\{a, b, c, d\}$, the probability that item $b$ is picked **best** is

$$
P(\text{best} = b) \;=\; \frac{\exp(\beta_b)}{\exp(\beta_a) + \exp(\beta_b) + \exp(\beta_c) + \exp(\beta_d)}
$$

After the best item is removed, the **worst** pick is modeled as a choice among the three remaining items, but with the utilities *flipped*. The least preferred item is the one with the lowest utility, which is the same as the highest $-\beta$:

$$
P(\text{worst} = c \mid \text{best} = b) \;=\; \frac{\exp(-\beta_c)}{\exp(-\beta_a) + \exp(-\beta_c) + \exp(-\beta_d)}
$$

So **each four-item task gives two MNL observations**: one best-pick from the four items, and one worst-pick from the three remaining items.

**Identifiability.** The soft-max does not change if we add the same constant to every $\beta$. For this reason, only $J - 1 = 9$ of the ten coefficients can be identified separately. We fix item 1 (MGTA 403) as the reference, so we set $\beta_1 = 0$ and we estimate $\beta_2, \ldots, \beta_{10}$ relative to it. Then every coefficient means "utility compared to MGTA 403."

```{python}
#| label: reshape-tasks

# For each four-item task store: the 4 item ids, the best pick, the worst pick.
SHOWN, BEST, WORST = [], [], []
for (_, _), g in four_item.groupby(["Record ID", "Task"]):
    SHOWN.append(g["Item"].values)
    BEST.append(int(g.loc[g["Response"] == 1, "Item"].iloc[0]))
    WORST.append(int(g.loc[g["Response"] == -1, "Item"].iloc[0]))

SHOWN = np.array(SHOWN)        # (n_tasks, 4) item ids in 1..10
BEST  = np.array(BEST)
WORST = np.array(WORST)
n_tasks = len(BEST)
print(f"Reshaped into {n_tasks} four-item tasks "
      f"(= {2 * n_tasks} MNL observations).")
```


## MNL via Maximum Likelihood

When we add the log-probabilities of the observed best and worst picks over all tasks, we get the log-likelihood

$$
\ell(\boldsymbol{\beta}) \;=\; \sum_{\text{tasks}} \Big[\, \beta_{j^*_b} \;-\; \log \!\!\!\sum_{j \in \text{shown}}\!\!\! \exp(\beta_j) \;-\; \beta_{j^*_w} \;-\; \log \!\!\!\!\!\sum_{j \in \text{shown} \setminus \{j^*_b\}}\!\!\!\! \exp(-\beta_j) \,\Big]
$$

where $j^*_b$ and $j^*_w$ are the items picked best and worst in that task. We write the negative log-likelihood as a function of the nine free coefficients (and we fix $\beta_1$ to $0$ inside the function). We use a log-sum-exp form for numerical stability, and we minimize the function.

```{python}
#| label: mle-fit

def make_beta(free9):
    """Insert the fixed reference (beta_1 = 0) into a length-11 vector
    indexed by item id; index 0 is unused so items map directly."""
    beta = np.zeros(11)
    beta[2:11] = free9          # items 2..10 are free; item 1 stays at 0
    return beta

def neg_loglik(free9):
    beta = make_beta(free9)
    u = beta[SHOWN]                                   # (n_tasks, 4)
    ll_best = beta[BEST] - np.log(np.exp(u).sum(axis=1))

    # Worst pick: remaining 3 items (shown != best), utilities flipped.
    mask = SHOWN != BEST[:, None]                     # True for remaining 3
    neg_u = np.where(mask, -beta[SHOWN], -np.inf)
    ll_worst = (-beta[WORST]) - np.log(np.exp(neg_u).sum(axis=1))

    return -(ll_best + ll_worst).sum()

res = minimize(neg_loglik, np.zeros(9), method="BFGS")
beta_hat = make_beta(res.x)
se_hat = np.zeros(11)
se_hat[2:11] = np.sqrt(np.diag(res.hess_inv))   # SEs from inverse Hessian

print(f"Converged: {res.success}")
print(f"Log-likelihood at optimum: {-res.fun:.2f}")
```

The standard errors come from the inverse of the (negative) Hessian at the optimum. The `scipy` BFGS optimizer returns this as `hess_inv`. To make the coefficients easier to interpret, we change them into **shares of preference** using a soft-max over all ten classes:

$$
\hat s_j \;=\; \frac{\exp(\hat\beta_j)}{\sum_{k=1}^{10} \exp(\hat\beta_k)}
$$

These shares add up to one, and they answer a concrete question: *if all ten classes were shown together on one screen, what fraction of the "best" votes would each class receive?*

```{python}

MGTA 495 is again at the top with about a 28% share, which is more than two times higher than any other class, and MGTA 444 Analytics Consulting is again last. The two methods agree completely at the top and the bottom, and they differ only in very small changes of order in the middle, where the coefficients (and so the shares) are not statistically different from each other; the standard errors of these middle classes overlap a lot. What the MNL adds is a *probabilistic* description: each item that is shown but not chosen also gives information, so the model uses each screen more completely than the counts score, which only looks at the picks. The story does not change, but the estimates become more precise.


## MNL via Bayesian Estimation

We can fit the same model with **Bayesian inference**. In this approach, $\boldsymbol{\beta}$ is treated as random, and the result is a full posterior distribution, not only a single point. The posterior is proportional to the likelihood multiplied by the prior,

$$
\pi(\boldsymbol{\beta} \mid \boldsymbol{y}) \;\propto\; L(\boldsymbol{\beta}) \cdot \pi(\boldsymbol{\beta})
$$

We use a weakly informative multivariate normal prior, $\boldsymbol{\beta} \sim N(\boldsymbol{0},\; 10 \cdot I)$. This is a standard deviation of about $3.2$ on each coefficient, which is wide enough to let the data decide, but still keeps the sampler stable. We cannot write the posterior in a closed form, so we draw samples from it with a **random-walk Metropolis-Hastings** sampler.

```{python}
#| label: mh-sampler

def log_post(free9):
    # log-likelihood (= -neg_loglik) plus log multivariate-normal prior
    log_prior = np.sum(-0.5 * free9**2 / 10.0)
    return -neg_loglik(free9) + log_prior

def metropolis(beta0, n_iter, step):
    K = len(beta0)
    draws = np.empty((n_iter, K))
    beta = beta0.copy()
    lp = log_post(beta)
    accept = 0
    for t in range(n_iter):
        beta_star = beta + np.random.normal(0, step, K)
        lp_star = log_post(beta_star)
        if np.log(np.random.rand()) < lp_star - lp:
            beta, lp = beta_star, lp_star
            accept += 1
        draws[t] = beta
    return draws, accept / n_iter

# Step size tuned to land acceptance in the 25-40% band (see below).
N_ITER, STEP = 20000, 0.07
draws, accept_rate = metropolis(res.x, N_ITER, STEP)

burn = N_ITER // 4               # discard first 25% as burn-in
post = draws[burn:]
print(f"Acceptance rate: {accept_rate:.3f}  (target 0.25-0.40)")
print(f"Iterations: {N_ITER}, burn-in: {burn}, kept: {len(post)}")
```

**Tuning the step size.** The random-walk step decides how far each proposal jumps. If the step is too small, the chain moves very slowly (the acceptance rate is near 1, but the chain almost does not move). If the step is too large, almost every proposal is rejected (the acceptance rate is near 0). A step of $0.07$ puts the acceptance rate inside the recommended band of 25–40%. If we make the step smaller, the acceptance rate goes up; if we make it larger, the acceptance rate goes down.

The traces look like fuzzy caterpillars around a stable level, and there is no slow drift. This is the visual sign that the chain has converged and is mixing well. Now we summarize the posterior. We compute the posterior **mean** of each coefficient, a 95% **credible interval** from the 2.5th and 97.5th percentiles of the draws, and the posterior **shares of preference**. To get the shares, we put every draw through the soft-max and then take the average.

```{python}
#| label: bayes-summary

# Prepend the fixed reference (beta_1 = 0) to every draw.
post_full = np.column_stack([np.zeros(len(post)), post])   # (n_kept, 10) for items 1..10

beta_mean = post_full.mean(axis=0)
beta_lo = np.percentile(post_full, 2.5, axis=0)
beta_hi = np.percentile(post_full, 97.5, axis=0)

# Posterior shares: soft-max each draw, then average across draws.
ex = np.exp(post_full)
share_draws = ex / ex.sum(axis=1, keepdims=True)
share_mean = share_draws.mean(axis=0)
share_lo = np.percentile(share_draws, 2.5, axis=0)
share_hi = np.percentile(share_draws, 97.5, axis=0)

bayes = pd.DataFrame({
    "item": range(1, 11),

    "post_mean_share": share_mean,
}).sort_values("post_mean_share", ascending=False)
bayes.round(3)
```

The three panels show almost the same result, and this is the main finding: **the methods agree very well at the top and at the bottom of the ranking.** MGTA 495 Marketing Analytics is the clear favorite in every method, and MGTA 444 Analytics Consulting, MGTA 451, and MGTA 403 are at the bottom in every method. The small disagreements appear only in the middle of the table. Classes such as SQL, the Capstone, and Large Data have very similar utilities, so a small amount of estimation noise can change their order between the methods. This is the normal behavior of any ranking: the two ends are stable, and the middle is sensitive.

What does the MNL add compared to the counts analysis? It describes the choice process directly. The counts score only records the picks, but the MNL also uses every *unchosen* item on a screen as information. For example, an item that loses to MGTA 495 on a strong screen tells the model something that the counts score ignores. The result is a consistent set of utilities that can also predict choices on screens that were never shown.

What does the Bayesian fit add compared to the MLE? It gives honest uncertainty on *derived* quantities. The shares of preference are a nonlinear transform of the coefficients, so to get a confidence interval on a share from the MLE, we need the delta method and some calculus. With the MCMC draws, it is very simple: we put each draw through the soft-max and read the 2.5th and 97.5th percentiles. The same simple method also works for any "what-if" quantity, for example market-share simulations or the effect of removing a class.


## Discussion

For a student or an advisor who only wants the main result: from 85 MSBA students and 680 best-worst screens, **MGTA 495 Marketing Analytics (Spring, Yavorsky) is the clear favorite**. It receives about a quarter of all "best" votes in a contest of ten classes, which is more than two times any other class. After it, there is a strong middle group, which includes Business Analytics, Customer Analytics, Large Data, the Capstone, and SQL. The least preferred classes are **Analytics Consulting, Marketing/Finance/Operations, and the AI Math/Programming course**. The small surprise is the size of the gap at the top: the most preferred elective is not only the highest, but it is higher by a large amount. All three methods agree on this result, so an advisor can be confident that it describes the students, and not only the method.

On the methodological side, each method is good for a different question:

- **Counts** is the right choice when we want a quick descriptive result, for example an executive dashboard or a first look. It does not need modeling assumptions, and it is easy to explain to anyone.
- **MLE MNL** is the right choice when we need rigorous point estimates with classical standard errors, and when we need one clear number for each item to report.
- **Bayesian MNL** is the right choice when we want uncertainty on *derived* quantities, for example shares of preference, market simulations, or "what if we add a new elective?" scenarios. It is also the right choice when we want to extend the model in a hierarchical way to estimate the preferences of each individual student.

There are also some caveats that we must remember. The respondents are a self-selected sample of MSBA students, so these preferences may not be the same for other cohorts or other programs. The preferences are also mixed together with things that the survey does not control: the professor of each class, the quarter when the class is offered, the prior coursework of the student, and the position of the student in the program can all affect the answers. For this reason, the design cannot separate "I like this subject" from "I like this professor."

If we had more time, the most natural next step is to use the anchor tasks that we put aside. These dual-response screens find an absolute zero point of preference, and with this point we could run a real *anchored* MaxDiff. Then we could say not only which classes are ranked higher than others, but also which classes the students actively like and which classes they only tolerate. After that, a hierarchical Bayesian extension could estimate one utility vector for each respondent. This would change a single program-level ranking into a tool for personalized course advising.
